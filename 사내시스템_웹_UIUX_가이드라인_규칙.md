# 현대백화점그룹 사내시스템 웹 UI/UX 가이드라인 규칙

> 출처: `2024.11.21. 그룹 사내시스템 웹 UI/UX 가이드라인` (PC 버전 기준, 85페이지)
> 개정: v5.1 (2024.11 브라우저 점유율 업데이트)
> 적용대상: 현대백화점그룹 사내시스템 (협력사용 시스템 포함) — 그룹웨어, 신열린인사, 백화점 H-RIS, 그린푸드 H-WIS, 백화점 위더스 등

---

## 표기 기준

- **[필수]** 반드시 지켜야 하는 사항. 미적용 시 서비스 시작 전 반드시 수정해야 하며, 품질 점검표에서 `YES`로 체크되어야 한다. (총 63개)
- **[권장]** 선택적으로 적용 가능한 사항. (총 50개)
- 검수 원칙: 상호 협의된 **필수 항목은 100% 조치 완료**를 원칙으로 한다.

---

## Ⅰ. 설계 기준 (기본 방향성)

### 1. 정보를 쉽게 찾을 수 있어야 한다
- 사이트 구조를 쉽게 알 수 있도록 한다 (탐색 → 검색 순서).
- 화면의 이동경로와 현재 위치를 알 수 있도록 설계한다. 네비게이션은 동일한 위치에 고정 노출하고, 현재 위치는 굵기·컬러·사이즈 등으로 구분한다.
- **3클릭·5초 법칙**: 3번 클릭 이내에 원하는 정보를 찾고 5초 이내에 이동 목적지를 알 수 있게 한다. 전체 메뉴 구조는 **3단계 이내**로 구성한다.
- 네비게이션을 일관성 있게 설계한다.

### 2. 직관적이고 일관성 있게 만들어 사용 편의성을 높인다
- 제공 정보와 기타 영역을 시각적으로 구분한다. 에러 메시지는 크기·색·위치를 고려해 눈에 띄게 만든다.
- 반응형 구현을 고려해 최대한 텍스트 기반으로 구성한다.
- 사용자 환경(OS·브라우저·스마트폰)에 따른 서비스 가능 여부를 사전 안내하고, 설치 진행 상황과 완료 결과(완료/실패/재설치)를 명확히 피드백한다.
- **불필요한 팝업창을 사용하지 않는다.** 공지 내용은 별도 게시판을 이용한다.
- 디자인은 통일성·규칙성을 갖는다. 색상은 **권고 6개 이내**로 사용한다.
- 동일 기능은 같은 아이콘을 재사용한다. 하이퍼링크는 다른 텍스트와 구분되게 표기한다.
- 화면 간 이동 없이 한 페이지에서 관련 업무를 처리하고, 절차의 중복을 제거·통합한다.

### 3. 모든 사용자가 이용할 수 있게 한다
- 다양한 디바이스 환경(해상도·PC사양·브라우저·폰트)을 고려한다.
- 웹 표준(W3C 권고안: 구조/표현/동작 분리)과 웹 호환성을 준수한다.
- 웹 호환성: 시장 점유율이 높은 **최소 3종 이상**의 브라우저에서 동일하게 표현되어야 한다.

### 4. 서비스를 안정적으로 이용할 수 있도록 한다
- 오류 메시지는 쉽게 인지되도록 하고, 상태 에러 코드는 사용자가 이해할 수 있는 메시지로 대체한다.
- 오류 발생 이유를 분명히 설명하고, 해결방안 및 IT 담당자 연락처를 오류 메시지에 기재한다.
- 이전 단계 복귀, 임시저장·복구 기능을 제공한다. 되돌릴 수 없는 경우 경고 메시지를 제공한다.

### 5. 사용자 경험 및 의견을 수용하여 서비스를 제공한다
- 사용자 행동 데이터(Log)를 축적·분석한다. (대시보드 제공 바람직)
- 정보·서비스에 대한 평가, 의견/댓글/이용후기 기능을 제공한다.
- 모니터링 담당자를 지정하고, 개선 사항을 적용하며, 개선 결과를 사용자에게 반드시 알린다.

## Ⅱ. 설계 가이드 체크리스트 (품질 점검표)

### 1. 브랜드 영역

#### A. 헤더 (pp.17~18)
1. **[권장]** 핵심적인 정보만을 간결하게 제공한다.
2. **[필수]** 로고와 사내시스템 명칭은 모든 사내시스템에서 좌측 상단에 배치한다.
3. **[권장]** 메인메뉴와 검색의 위치는 사내시스템의 목적에 적합하도록 우선 순위에 둔다.

#### B. CI (pp.18~21)
1. **[필수]** 워드마크만 단독 사용 불가하며, 워드마크와 로고타입이 혼용된 CI 형태를 사용한다.
2. **[필수]** CI의 가로·세로 크기를 임의로 왜곡할 수 없으며, 정비율로만 크기를 조정한다.
3. **[필수]** CI 바탕색은 기본 CI 배경 색상 외에 임의 변경이 불가하다.
4. **[필수]** CI 오용 사례를 항상 숙지하여 어떠한 경우라도 임의로 변경하여 사용하지 않는다.

### 2. 탐색

#### A. 메인 메뉴 (pp.22~23)
1. **[권장]** 링크는 우선 순위에 따라 배치한다. (중요·고빈도 링크일수록 좌측)
2. **[필수]** 링크의 활성화 상태가 명확히 구분되도록 표현한다.
3. **[권장]** 링크명은 이해하기 쉬운 용어를 사용한다.
4. **[권장]** 링크의 개수를 최소화한다.

#### B. 서브 메뉴 (pp.23~24)
1. **[권장]** 메뉴의 길이는 적절하게 제공한다. (아코디언 활용)
2. **[권장]** 이해가 동반되는 범위 내에서 자주 반복되는 단어는 삭제하고 표현한다.
3. **[필수]** 메뉴의 계층구조를 명확하게 구분하여 표현한다.
4. **[권장]** 기준이 애매한 메뉴명은 자세한 설명을 보여주는 말풍선 기능을 부여한다.

#### C. 탭 (pp.25~26)
1. **[필수]** 일관성 있는 표현방식을 사용한다.
2. **[필수]** 선택된 탭을 강조하여 표현한다. (색상 정보에만 의존하지 않는다)
3. **[필수]** 선택되지 않은 탭의 내용도 확인할 수 있게 표현한다. (과도한 흐림 금지)
4. **[권장]** 탭메뉴는 단일 행으로 제공한다.

#### D. 브래드크럼 (pp.27~28)
1. **[필수]** 일관된 위치(전역메뉴와 본문 사이)에 배치한다.
2. **[권장]** 적절한 기호(-, / 등)와 이미지를 활용해 경로 간 관계를 표시한다.
3. **[권장]** 가장 하위 수준(현재 페이지)의 경로를 구분하여 표현한다.

#### E. 사이트맵 (pp.28~29)
1. **[필수]** 사내시스템의 전체 메뉴를 포함한다.
2. **[필수]** 링크 목록을 사내시스템 구조에 맞게 계층화하여 표현한다.
3. **[필수]** 외부 사이트나 새 창으로 연결되는 링크를 분명하게 구분 표현한다.

### 3. 검색

#### A. 통합검색 (pp.30~32)
1. **[필수]** 통합검색 기능은 검색 상자와 검색버튼으로 구성한다.
2. **[필수]** 검색의 의미를 보편적으로 인식할 수 있는 레이블('검색')과 아이콘(돋보기)을 사용한다.
3. **[필수]** 통합검색은 모든 페이지에서 접근 가능해야 하며, 검색창은 상단 프레임 중앙 또는 우측에 배치한다.
4. **[권장]** 검색창 내에 검색어에 대한 안내(플레이스홀더)를 제공한다.
5. **[권장]** 조건(메뉴명칭·제목/내용·기간·작성자 등)을 설정하여 검색할 수 있도록 한다.

#### B. 검색결과 (pp.32~35)
1. **[필수]** 검색결과 페이지에 검색창을 유지한다.
2. **[권장]** 검색어와 일치하는 항목을 강조하고, 입력한 검색어와 결과 수를 표시한다.
3. **[권장]** 과거 검색 기록을 제공하여 바로 접근할 수 있는 기능을 제공한다.
4. **[권장]** 조회결과 가장 상단에 '메뉴검색' 조회결과를 표시한다.
5. **[필수]** 검색결과가 없는 경우, 결과를 명확하게 알리고 적절한 대안을 제시한다.

### 4. 게시판

#### A. 목록보기 (pp.36~38)
1. **[필수]** 중요한 항목은 목록의 상단에 배치한다.
2. **[권장]** 정렬기준을 명확하게 표현하여 사용자가 인지/추측 가능하게 한다.
3. **[권장]** 강조를 위한 시각적 표현을 과도하게 사용하지 않는다.
4. **[권장]** 긴 목록은 페이지 등으로 단위를 나누어 제공한다.
5. **[필수]** 규모가 큰 목록에 필터링 또는 검색 기능을 제공한다.

#### B. 게시물 작성/업로드 (pp.38~41)
1. **[필수]** 플레이스홀더를 이용하여 작성방법을 제공하지 않는다. (설명은 입력필드 상하단에 텍스트로)
2. **[필수]** 버튼을 용도 및 기능을 기준으로 구분하여 표현한다. ('취소'·'초기화'는 '등록'과 명확히 구분)
3. **[필수]** 저장/등록되지 않은 정보가 있을 때 페이지 이동 시 확인을 요청한다.
4. **[필수]** 등록 가능한 텍스트 및 파일서식/크기를 표시한다.

#### C. 상세내용 보기 (pp.41~42)
1. **[권장]** 제목을 다른 요소들과 구분되도록 표현한다.
2. **[권장]** 게시물 요소들 사이의 계층구조를 명확하게 구분한다.

#### D. 댓글 남기기/읽기 (pp.42~43)
1. **[필수]** 댓글입력에 사용하는 텍스트 입력필드는 충분히 크게 제공한다.
2. **[필수]** 작성자/관리자가 남긴 댓글을 다른 댓글과 분명하게 구분한다.
3. **[권장]** 페이징을 적절하게 사용한다.
4. **[권장]** 시스템 특성을 고려하여 게시물별·댓글별 좋아요(추천) 기능을 제공한다.

### 5. 데이터 작업

#### A. 데이터 입력 (pp.44~46)
1. **[필수]** 입력필드 주변에 레이블과 설명을 제공한다.
2. **[필수]** 필수입력 요소, 편집 가능한 요소를 분명하게 구분한다.
3. **[필수]** 입력필드의 크기를 충분하게 제공한다.
4. **[필수]** 입력값의 단위 및 범위를 표시한다.
5. **[필수]** 다수의 데이터를 일괄 업로드할 수 있는 파일서식과 기능을 제공한다.

#### B. 데이터 조회 (pp.47~48)
1. **[필수]** 조회결과에서 조회조건을 확인할 수 있도록 한다. (조건과 결과는 한 화면)
2. **[필수]** 데이터 조회 후 조회조건이 초기화되지 않도록 한다.
3. **[필수]** 데이터 조회 결과에 대한 정렬 기능을 제공한다.
4. **[권장]** 조회 건수·해당 페이지·금액 등 총괄적 의미의 내용을 표기한다.
5. **[권장]** 조회 버튼은 단축키를 이용하여 사용성을 높인다.
6. **[권장]** 자주 사용하는 조건을 디폴트로 제공한다 (최근 1개월 등).

#### C. 데이터 수정/삭제 (p.49)
1. **[필수]** 수정/삭제할 수 있는 내역을 알기 쉽게 구조화한다.
2. **[필수]** 일괄 선택기능을 제공한다.
3. **[필수]** 데이터 수정/삭제 버튼을 명확하게 구분한다. (아이콘 버튼은 마우스 오버 시 툴팁 제공)
4. **[필수]** 변경된 내용을 명확하게 표시·안내한다. (특히 삭제 시 재확인 메시지 제공)

#### D. 데이터 추출 (p.50)
1. **[필수]** 조회결과와 동일한 양식으로 추출되어야 한다.
2. **[필수]** 문자는 문자, 숫자는 숫자 서식으로 추출되어야 한다.

### 6. 안내

#### A. 모달 (pp.51~53)
1. **[필수]** 모달과 주변 영역을 분명하게 구분한다.
2. **[권장]** 스크롤 없이 정보를 확인할 수 있도록 만든다.
3. **[권장]** 단계적인 작업에는 모달을 사용하지 않는다. (별도 페이지 구성)
4. **[권장]** 하나의 모달이 다른 모달을 실행하지 않도록 한다.
5. **[권장]** 긍정적인 액션 버튼은 되도록 오른쪽에 배치한다. (반응형은 항상 오른쪽)
6. **[권장]** 행위 관련 버튼은 최대 2개까지 사용한다.

#### B. 확인 (pp.54~55)
1. **[권장]** 심각하거나 취소가 어려운 행동에 확인을 요청한다.
2. **[필수]** 상황에 대한 명확하고 간결한 설명을 제공한다.
3. **[필수]** 행동을 확정·취소할 수 있는 버튼을 모두 제공한다.
4. **[권장]** 버튼의 레이블에 행동을 구체적으로 설명한다.

#### C. 오류 (pp.55~56)
1. **[필수]** 빠르게 인지할 수 있도록 표현한다.
2. **[필수]** 발생한 오류 및 문제를 분명하게 설명한다. (ID/PW 오류 시 보안상 둘 다 확인 안내)
3. **[필수]** 오류를 수정할 수 있는 방법을 제공한다.

### 7. 지원/기타

#### A. 시스템에 대한 사용자 환경 (p.57)
1. **[권장]** 표준 해상도는 PC 기준 **1920 × 1080 (21인치)** 로 한다.
2. **[권장]** 데이터 & 스크롤 원칙을 따른다.
   - 중요·핵심 정보는 스크롤 없이 보이는 영역에 배치한다.
   - 화면에서 **가로 스크롤은 허용하지 않고**, 세로 스크롤은 필요 시 허용한다.
   - Tab: 가로 스크롤 불가, 세로 스크롤 제한적 허용.
   - Grid 내부: 가로·세로 스크롤 허용.

#### B. 크로스 브라우징 (p.58)

### 8. 컬러

#### A. 라이트 모드 (pp.65~66)
1. **[필수]** 일관성을 위하여 컬러의 구성 및 적용 범위를 규정하여 적용/응용한다.

3. **[필수]** 특별한 콘텐츠 표현을 위한 특수 컬러 사용은 예외로 둔다.
4. **[필수]** 시각적 피로도가 높은 컬러 사용은 자제한다. (빨간색은 필수 확인·오류·삭제 등 부정적 의미에 사용)
5. **[필수]** 텍스트 색상은 배경색과 충분한 대비를 둔다. (핵심 콘텐츠 명도 대비 **4.5:1 이상**)
6. **[권장]** 본문 영역 배경에는 패턴·텍스처 적용을 자제한다.
7. **[필수]** 시선 혼란을 막기 위해 색상 가지 수를 제한한다.

#### B. 다크 모드 (pp.68~71)
1. **[권장]** 너무 어둡지 않게 한다. (순수 검정 #000000 대신 다크 그레이 #121212 권장)
2. **[권장]** 적절한 대비를 갖게 한다. (회색 음영 + 낮은 채도)
3. **[권장]** 색상의 채도를 낮춘다. (파스텔 톤 등)
4. **[권장]** 올바른 '온(on)' 컬러를 사용한다. (순백 #FFFFFF 대신 밝은 회색)
5. **[권장]** 단순히 색상만 반전하지 않는다.
6. **[권장]** 라이트/다크 두 테마 각각의 색상 팔레트를 달리 만들어 적용한다.
7. **[권장]** 헤더·푸터·드롭다운 등 넓은 영역 요소의 표면 색상에도 다크모드를 적용한다.
8. **[필수]** 다크모드는 사용자가 설정·토글·버튼 등으로 직접 선택할 수 있어야 한다.

### 9. 단축키 (p.72)
